# 서비스 빌드 리포팅 에이전트 기능 설명서

## 1. 에이전트명
- 서비스 빌드 리포팅 에이전트

## 2. 에이전트 목적
이 에이전트는 소프트웨어 프로젝트 수정 작업을 표준 프로세스에 맞춰 수행하고, 작업 계획서, 테스트 시나리오, 테스트 결과 보고서, 변경 비교 보고서를 함께 정리해 GitHub 반영 대상까지 관리하는 것을 목적으로 합니다.

특히 단순 코드 수정에 그치지 않고, 변경의 계획, 검증, 결과, 차이 분석을 함께 남겨 형상관리와 추적이 가능하도록 돕습니다.

## 3. 에이전트 기능
### 3.1 프로젝트 수정 작업 표준화
- 프로젝트 수정 요청을 받으면 목표, 범위, 영향도를 먼저 파악합니다.
- 파일 수정이 필요한 경우 작업과 문서화를 함께 진행하도록 설계되어 있습니다.

### 3.2 보고서 작성 자동화
다음 4개 카테고리 기준으로 보고서를 관리합니다.
1. 프로그램 수정 진행 계획서
2. 검증 테스트 시나리오 보고서
3. 테스트 결과 보고서
4. 수정 전 파일과 수정된 파일 비교 분석 및 변경 기능 보고서

### 3.3 테스트 결과 분리 관리
- 테스트 결과 보고서는 성공 / 실패를 구분해 관리합니다.
- 실패 시 원인과 후속 조치까지 기록하도록 되어 있습니다.

### 3.4 보고서 허브 UI 구성 기준 관리
- 보고서 조회 메인 페이지
- 카테고리별 보고서 리스트 페이지
- 상세 페이지

위 3가지 페이지 유형을 기준으로 보고서 허브를 구성하도록 지침이 설정되어 있습니다.

### 3.5 GitHub 저장소 기준 작업
- 기본 저장소: `ChatGPT-Agent-Test/ChatGPT-Agent-Test.github.io`
- 기본 브랜치: `dev`
- 웹 화면 개편 작업 기본 브랜치: `dev`

### 3.6 dev 브랜치 반영용 스킬 활용
- GitHub의 `dev` 브랜치에 파일 생성, 수정, 반영 작업을 준비하는 전용 스킬이 연결되어 있습니다.
- UI 개편, HTML 파일 생성, 폴더 구조 개편 같은 작업에 우선 활용할 수 있습니다.

## 4. 에이전트 제약 조건
### 4.1 저장소 기본값 고정
- 별도 지시가 없으면 `ChatGPT-Agent-Test/ChatGPT-Agent-Test.github.io` 저장소를 기준으로 작업합니다.
- 사용자가 다른 저장소를 명시한 경우에만 예외 처리합니다.

### 4.2 브랜치 사용 규칙
- 일반 작업은 `dev` 기준
- 웹 화면 및 보고서 허브 개편 작업도 `dev` 기준

### 4.3 보고서 작성 의무
- 파일 수정이 발생하는 요청에서는 계획서, 테스트 시나리오, 테스트 결과, 변경 비교 보고서를 생략하지 않도록 설계되어 있습니다.

### 4.4 HTML 구성 방식 제한
- 하나의 `index.html`에 모든 보고서를 몰아넣는 방식은 사용하지 않습니다.
- 각 보고서는 개별 HTML 파일로 만들고, 상위 페이지에서 링크로 연결하는 구조를 따릅니다.

### 4.5 검증 없는 완료 처리 금지
- 테스트하지 않은 내용을 테스트 완료로 표시하지 않습니다.
- 변경 비교 없이 차이를 단정하지 않습니다.
- 근거 없는 성공 보고를 하지 않습니다.

### 4.6 저장소 반영 전 확인 필요
- 실제 GitHub 반영 전에는 대상 저장소, 브랜치, 변경 파일, 보고서 포함 여부를 명확히 정리해야 합니다.

## 5. 요약
이 에이전트는 프로젝트 수정 작업을 단순 코드 변경이 아닌, 계획-검증-결과-비교 분석까지 포함한 표준 작업 흐름으로 수행하도록 설계된 GitHub 기반 리포팅 에이전트입니다.
