# 시스템 로그 및 결함 관리 기능 검증 테스트 시나리오

## 테스트 목적
Supabase 미설정 상태에서도 사용자가 서비스 동작 상태를 명확히 이해할 수 있고, Supabase 설정 시 DB에 기록될 로그 구조와 ERROR 이상 결함 관리 기능이 화면에 정상 연결되는지 검증한다.

## 사전 조건
- 로컬 정적 서버가 `http://127.0.0.1:8000/`에서 실행 중이다.
- `assets/js/config.js`의 Supabase URL과 anon key는 placeholder 상태일 수 있다.
- 테스트는 Supabase 미설정 상태에서 로컬 대체 로그를 기준으로 수행한다.
- 실제 운영 DB 반영은 `supabase/schema.sql` 실행 후 Supabase 설정값을 주입해야 한다.

## 시나리오 1. 메인 화면 시스템 상태 표시
절차:
1. `http://127.0.0.1:8000/`에 접속한다.
2. 시스템 상태 영역을 확인한다.
3. 시스템 로그 관리 링크가 표시되는지 확인한다.

기대 결과:
- 운영 상태가 `제한 동작`으로 표시된다.
- DB 로그 저장이 불가하다는 정보가 표시된다.
- 시스템 로그 관리 페이지로 이동하는 링크가 표시된다.

## 시나리오 2. 시스템 로그 관리 화면 기본 로딩
절차:
1. `http://127.0.0.1:8000/pages/system-logs.html`에 접속한다.
2. 결함 우선 요약, 로그 레벨 필터, 우선 확인 결함, 전체 로그 영역을 확인한다.

기대 결과:
- Supabase 미설정 상태가 로컬 대체 로그 조회 상태로 표시된다.
- 로그 레벨 필터가 `전체`, `TRACE`, `DEBUG`, `INFO`, `WARN`, `ERROR`, `FATAL`, `CRITICAL`로 표시된다.
- ERROR 이상 로그가 없을 경우 우선 확인 결함 없음 메시지가 표시된다.

## 시나리오 3. ERROR 이상 결함 우선 조회
절차:
1. `http://127.0.0.1:8000/pages/system-logs.html?seedTestLogs=1`에 접속한다.
2. 검증용 `ERROR`, `CRITICAL` 로그가 생성되는지 확인한다.
3. 우선 확인 결함 영역과 요약 카드의 값을 확인한다.

기대 결과:
- 미해결 ERROR 이상 건수가 증가한다.
- 미확인 ERROR 이상 건수가 증가한다.
- `CRITICAL` 로그가 우선 확인 결함에 표시된다.

## 시나리오 4. 결함 확인 및 해결 처리
절차:
1. 우선 확인 결함 카드에서 `확인 처리` 버튼을 클릭한다.
2. 해결 메모를 입력한 뒤 `해결 처리` 버튼을 클릭한다.
3. 요약 카드와 결함 상태 문구를 확인한다.

기대 결과:
- 확인 처리 후 상태가 확인 완료로 바뀐다.
- 해결 처리 후 상태가 해결 완료로 바뀐다.
- 미해결/미확인 요약 수치가 감소한다.

## 시나리오 5. 로그 레벨 필터
절차:
1. 시스템 로그 관리 화면에서 `WARN` 필터를 클릭한다.
2. `ERROR` 필터를 클릭한다.
3. `전체` 필터로 되돌린다.

기대 결과:
- 선택한 레벨에 해당하는 로그만 전체 로그 목록에 표시된다.
- 필터 전환 후 화면이 오류 없이 다시 렌더링된다.

## 시나리오 6. 보고서 연결 검증
절차:
1. `data/reports.json`을 파싱한다.
2. 신규 보고서 ID `rpt_20260521_005`, `rpt_20260521_006`이 연결되었는지 확인한다.
3. 모든 `file_path`가 실제 파일과 일치하는지 확인한다.
4. 모든 `detail_page_path`가 `pages/report-detail.html?reportId={report_id}` 규칙과 일치하는지 확인한다.

기대 결과:
- JSON 파싱 오류가 없다.
- 신규 보고서가 페이지 데이터에 연결되어 있다.
- 누락된 Markdown 파일이 없다.
- 상세 페이지 경로 규칙 위반이 없다.
