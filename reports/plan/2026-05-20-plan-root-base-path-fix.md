# 작업 수행 계획서

## 작업 개요
- 작업명: GitHub Pages 루트 배포 경로 오인 원인 분석 및 수정 계획
- 작업 유형: 신규 작업
- 작성 일시: 2026-05-20

## 배경
개발 작업 리포트 허브 메인 페이지는 정상적으로 렌더링되지만, 실제 배포 화면에서는 시스템 상태, 보고서 바로가기, 보고서 목록이 모두 로딩 실패 상태로 표시되었다.

확인 결과, GitHub Pages 캐시 문제를 해소한 뒤에도 데이터 로딩 실패가 지속되었고, 원인은 정적 자산 경로 계산 로직과 실제 배포 구조의 불일치 가능성으로 좁혀졌다.

## 확인된 핵심 원인
- 현재 배포 저장소는 `ChatGPT-Agent-Test.github.io` 형태의 사용자 페이지 저장소다.
- 이 형태는 GitHub Pages에서 루트 경로(`/`)로 서비스되는 것이 기본이다.
- 그러나 프런트엔드 설정 파일에서 저장소명을 별도 프로젝트 사이트처럼 해석할 수 있는 값으로 유지하고 있었다.
- 그 결과 데이터 파일과 상세 페이지 경로가 실제 루트 기준이 아니라 하위 경로 기준으로 계산될 가능성이 생겼다.
- 메인 페이지 렌더링은 성공했지만, `data/reports.json` 및 관련 리소스 조회가 실패하면서 전체 초기화가 catch로 빠졌다.

## 작업 목표
1. 루트 배포 구조를 기준으로 경로 계산 규칙을 명확히 정리한다.
2. 왜 화면은 보이지만 데이터만 실패했는지 재현 가능한 형태로 문서화한다.
3. 이후 수정 시 참고할 수 있도록 테스트 시나리오와 검증 기준을 남긴다.
4. 이번 원인 분석을 기존 문서 덮어쓰기 없이 신규 보고서 세트로 남긴다.

## 수행 범위
- GitHub Pages 배포 구조 해석
- 프런트엔드 경로 계산 로직 영향 분석
- 실패 증상과 원인 연결
- 수정 포인트 및 검증 기준 정리

## 제외 범위
- 실제 코드 수정 반영
- Supabase 실연동 값 주입
- 배포 워크플로우 자체 변경

## 예상 수정 포인트
- `assets/js/config.js`
  - `REPOSITORY_NAME` 해석 방식 재검토
  - 사용자 페이지 저장소(`*.github.io`)일 때 `BASE_PATH`를 빈 값으로 유지하도록 보정
- 경로 의존 함수
  - `buildAppPath`
  - 상세 페이지 링크 생성 로직

## 검증 기준
- 루트 주소 `https://chatgpt-agent-test.github.io/` 에서 `data/reports.json` 요청이 성공해야 한다.
- 메인 화면의 시스템 상태, 요약 카드, 보고서 목록이 정상 표출되어야 한다.
- 상세 페이지 링크가 루트 기준으로 정상 이동해야 한다.
- 기존 문서는 유지되고, 이번 작업 결과는 신규 문서로 분리되어야 한다.

## 결론
이번 작업은 기존 보고서 업데이트가 아니라, 배포 경로 오인이라는 새로운 장애 원인 분석 작업이므로 신규 문서 세트로 관리한다.
